# Migrate workspace memory from `.claude-memory-compiler` to `llm-wiki-memory` (with custom Jira tree)

> After approval, copy to `~/repos/.claude/plans/pending/migrate_to_llm_wiki_memory.md` (no Jira issue — name per user request) and delete this autogenerated file, per `plans-lifecycle.md`.

## Context

The current memory system at `/Users/developer/repos/.claude-memory-compiler/` (Python + Claude Agent SDK, three hooks in `repos/.claude/settings.local.json`) compiles daily session logs into a small wiki of concepts/connections at `~/repos/.claude-memory-compiler/knowledge/`. It works, but it doesn't support semantic retrieval, has no Jira-issue topology, and ships no per-folder index files. We're switching to [`llm-wiki-memory`](https://github.com/ctxr-dev/llm-wiki-memory) — a project-local, Node-based system with Xenova on-device embedding search, a typed atom schema, and stable reusable internals (`scripts/lib/wiki-store.mjs`) we can call from custom hooks.

On top of vanilla `llm-wiki-memory`, we're adding:

1. A custom **Jira issues tree** at `wiki/issues/JIRA/<PREFIX>/<N÷1000>/<(N%1000)÷10>/<N%10>/<PREFIX>-<N>.md` — each leaf folder holds exactly one issue's knowledge file plus its plan files (in `pending|in-progress|done|archived/` subfolders).
2. A **custom hook** `~/repos/.hooks/jira-sync.mjs` that runs on SessionEnd / PreCompact / PostCompact / PostToolUse(ExitPlanMode) to: scan Jira keys in the transcript, append findings to issue knowledge files, sync checkbox state, move plan files between lifecycle folders, parse `reason:<key>:<comment>` tags, and regenerate every touched `index.md`.
3. A **SessionStart context warm-up** so we don't lose the compiler's auto-injection of recent context.
4. Full **legacy import** of compiler knowledge + dailies, plus migration of `.claude/{notes,plans,memory}` into the new wiki.

The compiler stays on disk (with hooks removed) until the new system is confirmed stable — then everything compiler-related is deleted.

## Locked design decisions

- **Install scope**: single shared wiki at `~/repos/.llm-wiki-memory/` (cross-repo).
- **Embedding model**: `Xenova/bge-large-en-v1.5` (340 MB, set via `.env` before first recall).
- **Wiki git**: `git init` inside `~/repos/.llm-wiki-memory/wiki/` (local only, no remote). Powers the hook's session-touch detection via `git diff`.
- **`/repos-2/.claude-memory-compiler/`**: delete after backup. Out of scope.
- **Jira API**: NO direct REST integration. The session-active Claude fetches via the Atlassian MCP and writes results into knowledge files. The hook is purely local.
- **Connection file (1 file)**: migrate as `wiki/knowledge/cross-cutting/<slug>.md` with `atom_type: connection` — no custom `connections/` folder.
- **Index.md regeneration**: per-session for ALL levels touched (walk up to `wiki/`).
- **Hook language**: `.mjs` (Node), so it can `import` `wiki-store.mjs` directly.
- **Lifecycle moves**: automatic. Hook moves files AND updates `status:` frontmatter. `archived` is manual-only.
- **Non-Jira plans**: `wiki/plans/pending|in-progress|done|archived/<slug>.plan.md` (lifecycle folders mirror the existing rule).
- **Tool choice**: Claude does write-time (drafting frontmatter, distilling findings, classifying); Xenova does read-time (semantic recall).
- **Planning convention**: This plan's own structure (Context → Locked decisions → max-granularity action items grouped by phase → Edge case analysis → Code review cycle → Critical files → Verification) becomes a first-class rule in the new system (`~/repos/.claude/rules/plan-format.md`). All future plans — Jira-linked and not — must follow this shape. The Jira hook validates new plan files against it and refuses to lifecycle-promote anything that doesn't conform.

### 📝 Surface findings from B++++ worth knowing

1. **Node's `vm.createContext` exposes some host built-ins by default** — `console`, `queueMicrotask`, `setTimeout`, `setInterval`, `setImmediate`, and their `clear*` counterparts. We explicitly set them to `undefined` in the sandbox object. `globalThis` IS the sandbox itself (standard JS), not a leak.
2. **Sandbox-returned objects have a different `Object` prototype than the host realm**, so `assert.deepStrictEqual` on a `from_path` result fails even when the values look identical. Tests compare property-by-property to dodge this; production code should be aware if it ever needs to compare sandbox results.
3. **`parseInt` is NOT whitelisted in the sandbox** (it's a global function in JS, not on `Number`). Use `Number(...)` for integer coercion inside compilers. Documented in PROTOCOL.md.
4. **Forward precedence** is `to_path_file > to_path > path_template`. The validator's superRefine enforces exactly-one, so users can't accidentally have two competing forward mechanisms.

### ⚠️ Critical deviations discovered during Phase B

Two findings invalidate parts of the original plan and require user decision before Phase C:

1. **`index.md` is auto-generated by the skill** at every folder. The bootstrap-shipped root `index.md` has frontmatter `generator: "skill-llm-wiki/v1"` and is regenerated on writes. **Impact:** Phase E (custom index regenerator) is largely unnecessary for the 5 native categories — but the skill won't index our custom `issues/` tree (see #2). The Jira hook still needs to generate `index.md` for files outside the native layout.

2. **The skill enforces a CLOSED set of top-level categories.** Source at `scripts/lib/wiki-store.mjs:222-230`:
   - `CATEGORIES = ["knowledge", "self_improvement", "plans", "investigations", "daily"]` (hardcoded)
   - `assertKnownSlot()` throws `WikiStoreUnavailable` if you try to write to any other top-level
   - `placementDirForMeta(category, meta)` computes path from frontmatter facets: `knowledge → area/atom_type/`, `plans → area/`, `investigations → area/`, etc.
   - Writing under `wiki/issues/JIRA/.../` directly bypasses ALL the skill's machinery: validate fails, search ignores them, no auto-index.

   This means our planned Jira tree `wiki/issues/JIRA/<PREFIX>/<k>/<h>/<u>/` cannot be created via `saveDocument`/`writeMemory` without modification.

**Decision required before Phase C** — pick one approach for hosting Jira artifacts:

- **Option A: Patch the skill source** to add `issues` as a 6th category and add custom placement facets `[prefix, kthousand, khundred, kunit]`. Keeps our exact tree shape. Cost: maintenance burden every time we `git pull` upstream.
- **Option B: Map Jira artifacts onto existing categories.** Knowledge file → `investigations/jira-<PREFIX>-<N>/<PREFIX>-<N>.md`; plan files → `plans/jira-<PREFIX>-<N>/<PREFIX>-<N>-<slug>.plan.md`. Keeps full semantic recall + auto-index + validate. Loses the 129/95/7 nested formula (each issue gets its own flat `area` folder).
- **Option C: Write Jira artifacts outside the skill** at `wiki/issues/...` using direct `fs.writeFileSync`. Keeps our exact tree. Custom Jira hook does its own indexing + we figure out semantic recall separately (e.g. our own embedding pipeline, or skip semantic for Jira). Validate output likely shows warnings/errors.
- **Option D: Hybrid.** Real Jira data goes into `investigations/jira-<PREFIX>-<N>/` (Option B path, for recall + validate). Our `wiki/issues/JIRA/<PREFIX>/<k>/<h>/<u>/` tree is a parallel "human-readable view" of pointer files (symlinks or markdown links) generated by our hook. More moving parts; needs index sync.
