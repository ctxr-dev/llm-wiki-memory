# Cron PATH fix + provider-unavailable observability (exit 69 + escalation)

## Context

The hourly maintenance job (launchd plist on macOS / crontab wrapper on Linux, generated by `bootstrap.sh --schedule daily`) ran with launchd/cron's minimal PATH, hiding the provider CLIs: compile aborted `LLMProviderUnavailable` with exit 0, cron-job recorded ok, escalations never fired, cron-health stayed healthy — the failure self-reported never. Confirmed live on tick `2026-06-04T13:00:02Z` (2 daily docs silently unpromoted).

## Locked decisions

| Decision | Choice |
|---|---|
| PATH source | Hybrid: live `$PATH` ∪ `dirname(execPath)` ∪ curated dirs, deduped, first-occurrence wins (`scripts/lib/cron-path.mjs`) |
| Where applied | Both layers: bootstrap bakes into plist/wrapper AND `llm.mjs spawnCapture` augments at runtime via the same helper |
| Compile signal | Exit 69 (`EX_UNAVAILABLE`) on the per-atom `LLMProviderUnavailable`/`WikiStoreUnavailable` abort; cron-job special-cases 69 and still runs consolidate |
| Health flip | Immediate failed attempt, self-clearing on next good tick |
| Escalation | Synthetic entities `system:compile-llm-providers` / `system:consolidate-llm-providers` (pass names `compile-promote` / `consolidate-llm`) through the existing machinery; reuse `consolidate.escalateAfterAttempts`; success ticks resolve episodes. Tail-first excerpt so ENOENT vs timeout open distinct episodes within the 80-char signature window |
| Consolidate LLM-skip | In scope (failure entity only on real runs with `llmRequested && !llm`; `--no-llm` contributes nothing); attempt stays ok:true |
| Bookkeeping placement | Inside `finish()` so early-return paths (consolidate hard-fail) still record the compile streak (review finding, fixed) |
| Off-cycle occurrence filter | Synthetic-only ticks append occurrences only for synthetic-pass signatures (prevents hourly churn of pending consolidate episodes) |
