# Docs & README style

Conventions for `README.md`, `ARCHITECTURE.md`, and everything under `docs/`. Each came
from explicit user feedback — treat them as fixed, not preferences.

## Structure & copy

- **Install before Highlights.** A reader must reach install instructions without
  scrolling past marketing.
- **The header sells; the sections explain.** Open with the user's pain and the outcome,
  name the supported clients explicitly (Claude Code, Cursor, Codex — these are also the
  search terms), close with a call to action. Detail vocabulary (write-gate, consolidation
  passes) stays in its own sections.
- **No dash punctuation in header copy** — no em dashes; reword instead. Prefer plain
  words over compound hyphens where natural ("versioned in git", not "git-versioned").
- **Define every internal shorthand in plain English BEFORE first use.** Levels / layers /
  tiers (e.g. write-gate enforcement layers) get a short table: what each is, why it
  exists, how they reconcile. Undefined jargon reads as noise.
- **Phrase counts count-free** where possible ("the documented top-level keys", "the full
  suite"). When a count is unavoidable (the tests badge), updating it is part of whatever
  change moved the number — it lives in exactly TWO places: the shields badge at the top
  and the Testing section.
- **`AI-INSTALL-PROMPT.md` is the canonical agent install/update procedure.** The README
  Install section carries only the copyable one-liner that points at it (raw URL +
  local `@` path) plus the human-facing manual command blocks. Procedure changes go to
  the canonical file FIRST; never let the README grow a second copy of the prompt.

## Diagrams

- Mermaid only, vertical (`flowchart TD`) — never ASCII art, never `LR` for pipeline
  diagrams.
- Quote any node label containing `/`, `%`, or other specials; prefer rectangles over
  cylinders for filesystem-path labels.
- Dotted edges with labels use the pipe form `-.->|label|`, and the label is quoted when
  it contains a dot (`"cli.mjs redistill"`). The inline `-. label .->` form breaks
  Mermaid's lexer when the label contains a dot.

## Badges

- GitHub clone/traffic counts are admin-private; the stargazers badge is the public
  popularity proxy. Don't promise a clone-count badge.

## Visual theme (gentle cyberpunk)

The README is themed after the org profile's Cyberpunk 2077 look. GitHub markdown has no
CSS, so the theme lives in exactly these tokens — reuse them, don't invent new ones:

- Palette: signature yellow `FCEE0A`, cyan `00F0FF`, alert red `FF003C`, neon mint
  `00FF9F` (passing/health only), near-black `0D0D14`.
- **Badge readability scheme (applies to EVERY badge):** neon goes on the LABEL side
  (`labelColor=<neon>`, shields auto-renders black text on light neon), the MESSAGE side
  is dark (`color=0D0D14`, white text). Never put white/light text on a neon background.
  Header badges: `style=for-the-badge`, UPPERCASE, two rows (authority row first: tests,
  node, license, MCP). Capability rows keep brand colors as the label side.
- Highlights: numbered `flat-square` chips (`01`..`NN`), ALL yellow labels (`FCEE0A`) +
  dark message; each chip on its OWN line, blank line, then a bold hook sentence + prose.
- **No `<style>` blocks, ever:** GitHub's sanitizer strips `<style>` elements AND
  `style=` attributes from README markdown — global CSS (e.g. for tables) is impossible.
  The per-element `<img>` frames below are the only sanitization-proof mechanism; they
  are regenerated by script, which is what makes them behave like a global rule.
- Hero banner = `docs/assets/banner.svg`, the ONE sanctioned repo-local image (custom
  art is impossible via external generators, and GitHub strips data URIs). 1200×240:
  dark teal vignette (`04090C`/`071A20`) behind a NEURON WEB (user directives 2026-06-04,
  synapse/color/title directives revised later the same day: no microschema/PCB traces —
  brain-neuron web, uniform star-shaped cells; cells randomly distributed, never
  grid-ordered; synapses are smooth WANDERING CURVES, never straight and never a regular
  sine — direction changes vary). Generated, not hand-drawn: rejection-sampled random
  somas (min-distance spacing, title band kept clear) with 6–8 jagged radiating
  dendrites each (some fork at the tip — dendrites stay jagged; only synapses are
  curved), synapse edges as Catmull-Rom curves through randomized waypoints
  (perpendicular offsets of random magnitude, sign mostly alternating but sometimes
  repeating; waypoints landing in the title band are pushed vertically out of it),
  3-nearest-neighbour edges PLUS ~12 long-range links between non-neighbours (240–540px,
  degree-capped, fainter at stroke-opacity 0.55), faint bokeh dots, the whole mesh in
  ONE group at ~35% opacity; ~34 signal particles travel the edges via SMIL
  animateMotion+mpath (staggered durs/begins, ~⅓ reversed via keyPoints="1;0").
  Interface cyan `5EF6FF` / teal `2BD9E6` / gold accent `FCEE0A` — NO red/magenta cells
  (a few gold accent neurons via the `somaGold` gradient); glitch magenta `FF2A6D`
  survives ONLY as the title's chromatic-aberration ghost layer. Title is vertically
  centered (baseline y=139 for the 54px wordmark in the 240px canvas) with a periodic
  neon-flicker blink (SMIL opacity animation on the title group). RAM widget (lit cells
  solid, empty cells genuinely transparent strokes), blinking MEM.SYS cursor. KNOWN TRADEOFF:
  renders on github.com but 404s in IDE previews until committed/pushed — preview by
  opening the SVG file itself. Keep all animation SMIL-only (no CSS/script inside the
  SVG).
- Section/divider lines keep the SHARED 3-stop gradient
  `color=0:00F0FF,50:FCEE0A,100:FF003C` (verified: all three stops render).
- **Divider placement is a strict rule, not a vibe:** gradient lines exist in exactly two
  roles. (1) SECTION lines, `height=4`, appear ONLY on the line immediately after a
  top-level `## ` header — no blank line between header and line (hug the header), one
  blank line after. Never at a section's end, never free-floating. (2) TABLE frames,
  `height=1` (hairline), hug their table: frame line, ONE blank line (parser-required —
  a GFM table directly after an HTML line gets swallowed into the HTML block), table
  rows, then the bottom frame DIRECTLY on the next line after the last row (no blank).
  The blank-line rules ARE the margin control.
- Every divider/frame `<img>` carries an EXPLICIT `height="N"` attribute matching the
  URL height: with only `width="100%"`, browsers aspect-scale the SVG and the rendered
  thickness varies by layout context (the top/bottom table frames once rendered 2×
  apart because of this).
- **RELATIVE repo-asset links, ALWAYS (user directive 2026-06-04).** Decorative images
  (banner, lines, custom art) live in `docs/assets/` and are referenced relatively
  (`docs/assets/line.svg`) — never absolute raw URLs, never external generator services
  for repo-owned art. External https is permitted only for genuinely dynamic badges
  (shields tests/stars). Accepted tradeoff: relative assets 404 in some IDE previews
  until committed/pushed — preview by opening the SVG file directly. Never use services
  that wrap content in SVG `foreignObject` (svg-banners et al.).
- **Relative assets use MARKDOWN image syntax (`![](docs/assets/x.svg)`), never HTML
  `<img>`:** several markdown previewers rewrite relative paths only for markdown-syntax
  images and leave raw HTML `src` unresolved (invisible/broken until pushed). Markdown
  syntax also sidesteps the spec-invalid `width="100%"` attribute entirely.
- **Line thickness via intrinsic aspect, not attributes:** the over-wide asset is clamped
  to container width by GitHub's `max-width:100%`, and height scales proportionally —
  `line-bold.svg` (2000×9 ≈ 4px at GitHub's ~888px content width) for section lines,
  `line-thin.svg` (2000×2 ≈ 1px) for table frames. Thickness scales gently with viewport,
  and the bold:thin ratio is constant everywhere.
- Mermaid: every flowchart starts with the shared init directive — `theme: base`,
  `flowchart.curve: linear` (straight edges, never curved), `lineColor #00B8C4`, node
  fill `#0D0D14` with `#FCEE0A` text/border, `edgeLabelBackground #0D0D14`, `textColor
  #00B8C4`. True gradient strokes are NOT possible in Mermaid; do not fake them.
- Tables cannot take CSS borders on GitHub; the gradient frame (divider img above and
  below) is the sanctioned substitute.
- `<samp>` for the terminal-feel punch line in the header; nowhere else.
- Keep it GENTLE: theme decorates the existing copy; it never replaces information, and
  every shields URL must return SVG (curl-check) before shipping.
